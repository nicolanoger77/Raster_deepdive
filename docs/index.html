## Task 1: Raster Approach

# Create a template raster based on the extent of the cantons
r_template <- rast(kantone, resolution = 100)

# Rasterize cantons: assign the canton name to each pixel
r_kantone <- rasterize(kantone, r_template, field = "name")

# Rasterize forest: assign 1 to forest pixels, 0 to background
r_wald <- rasterize(wald, r_template, field = 1, background = 0)
names(r_wald) <- "wald_pixel_count"

# Count forest pixels per canton using zonal statistics
wald_pro_kanton <- zonal(r_wald, r_kantone, fun = "sum", na.rm = TRUE)

# Calculate forest area in square meters (1 pixel = 100m * 100m = 10,000 m^2)
wald_pro_kanton <- wald_pro_kanton |> 
  mutate(wald_area = wald_pixel_count * 10000)

# Calculate total canton area (using the fast vector method) and join
kantone$kanton_area <- expanse(kantone)

kanton_frac <- kantone |>
  as.data.frame() |> 
  group_by(name) |>
  summarise(kanton_area = sum(kanton_area)) |>
  left_join(wald_pro_kanton, by = "name") |> 
  mutate(wald_frac = as.numeric(wald_area / kanton_area))

# End and log runtime
terra_time <- tictoc::toc(quiet = TRUE, log = TRUE)
terra_time <- as.numeric(terra_time$toc - terra_time$tic)

Results

The raster approaches with terra and GDAL produce the same forest shares as the vector computations (sf/DuckDB). The canton order and percentages are consistent, so the results are methodologically identical.

Discussion

Raster processing was much faster than the vector calculations because the raster data are smaller here and the evaluation is essentially a simple pixel count. The repeated OOM issues with terra were unexpected and even crashed my device, which suggests high memory demand during rasterization or intermediate steps despite the smaller data footprint. Overall, the comparison confirms that raster is more efficient here, but terra was unstable in my setup.
